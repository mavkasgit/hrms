# Дизайн: Ручное резервное копирование и восстановление БД

## Дата: 2026-04-30
## Тема: backups
## Статус: Одобрено пользователем

---

## 1. Общая архитектура (Подход A: Docker Exec)

Backend определяет имя Docker-контейнера Postgres по переменной окружения `POSTGRES_CONTAINER_NAME` (или выводит из `DATABASE_URL` + hardcoded mapping: `hrms_dev` → `hrms-postgres`, `hrms_test` → `hrms-postgres-test`, `hrms_prod` → `hrms-postgres-prod`).

Все операции выполняются через `docker exec <container> pg_dump ...` / `pg_restore ...`.

**Новые volume для prod/test compose:** Docker socket `/var/run/docker.sock:/var/run/docker.sock:ro` у backend-контейнера, чтобы он мог управлять соседним postgres-контейнером.

**Директория бэкапов:** `backend/data/backups/` — монтируется как volume, доступна локально в dev.

## 2. Backend API (`/api/backups`)

Новый роутер `app/api/backups.py`. Доступ: `current_user == "admin"` (как в templates.py).

| Метод | Эндпоинт | Описание |
|-------|----------|----------|
| POST | `/api/backups` | Создать бэкап текущей БД. Имя: `backup_<db_name>_<timestamp>.dump`. Возвращает метаданные созданного файла. |
| GET | `/api/backups` | Список всех бэкапов: имя, размер, дата создания, имя БД. |
| GET | `/api/backups/{filename}/download` | Скачать файл бэкапа. |
| POST | `/api/backups/{filename}/preview` | **Создать превью статистики** из существующего бэкапа. |
| POST | `/api/backups/upload-preview` | **Загрузить .dump файл и получить превью статистики.** |
| POST | `/api/backups/{filename}/restore` | Восстановить текущую БД из выбранного бэкапа, затем `alembic upgrade head`. |
| POST | `/api/backups/upload-restore` | Загрузить .dump файл и восстановить текущую БД из него, затем `alembic upgrade head`. |

**Структура Preview Response:**
```json
{
  "source_db": "hrms_prod",
  "backup_timestamp": "2026-04-30T12:34:56",
  "tables": {
    "employees": 45,
    "orders": 128,
    "vacations": 67,
    "departments": 8,
    "sick_leaves": 12,
    "order_types": 15
  }
}
```

## 3. Flow превью статистики

**Шаги backend для `POST /api/backups/{filename}/preview`:**
1. Создать временную БД: `CREATE DATABASE hrms_preview_<uuid>` (через `docker exec` `psql`).
2. Восстановить бэкап во временную БД: `pg_restore -d hrms_preview_<uuid> <file>` (без `--clean`, это чистая БД).
3. Подключиться к временной БД и выполнить `SELECT count(*) FROM employees, orders, vacations, ...`
4. Собрать статистику в JSON.
5. Удалить временную БД: `DROP DATABASE hrms_preview_<uuid>`.
6. Вернуть JSON-ответ.

**Ошибки:** если бэкап поврежден или restore падает — возвращаем 422 с описанием ошибки от `pg_restore`.

## 4. Flow восстановления (с миграциями)

**Шаги backend для `POST /api/backups/{filename}/restore`:**
1. **Валидация:** проверить, что пользователь ввёл правильное имя БД в теле запроса.
2. **Закрыть соединения:**
   - `engine.dispose()` — закрыть все соединения из пула SQLAlchemy.
   - `SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE datname = '<db>' AND pid <> pg_backend_pid()`
3. **Восстановление:** `pg_restore --clean --if-exists -d <db> <file>`
4. **Миграции:** `alembic upgrade head` — применить все миграции, которые появились после создания бэкапа. Это гарантирует, что схема БД соответствует текущему коду приложения.
5. **Пересоздать engine:** `create_async_engine(...)` снова, чтобы новые соединения использовали актуальную схему.
6. Вернуть 200 с сообщением об успехе.

**Важно:** если `alembic upgrade head` падает (например, бэкап настолько старый, что миграции несовместимы), восстановление считается неуспешным, и пользователь получает ошибку с логами alembic.

---

## 5. Совместимость с миграциями

**Проблема:** `pg_dump` сохраняет схему + данные на момент создания бэкапа. Если с тех пор:
- Появились новые миграции (новые таблицы, колонки, индексы) → код ожидает их, а в восстановленной БД их нет.
- Удалились старые колонки/таблицы → в восстановленной БД они остались, но код о них не знает (обычно не критично).

**Решение:** после `pg_restore` всегда выполнять `alembic upgrade head`. Это:
- Создаст недостающие таблицы и колонки.
- Обновит существующие структуры.
- Не затронет данные, которые уже есть в восстановленной БД.

**Ограничение:** если миграции содержат **data migrations** (перенос/трансформация данных), которые зависят от промежуточных состояний, `alembic upgrade head` может упасть на очень старом бэкапе. Рекомендация: бэкапы старше N версий миграций считать "устаревшими" (предупреждение в UI).

**Что НЕ нужно делать:**
- Не нужно "вливать" миграции в сам `.dump` файл — `pg_restore` полностью заменяет БД, и `alembic upgrade head` позаботится о схеме.

## 6. Frontend UI (`/settings/backups`)

Новый подраздел в Settings, как `/settings/holidays`.

**Страница содержит:**
- **Заголовок** "Резервное копирование БД"
- **Кнопка "Создать бэкап"** — вызывает POST /api/backups, обновляет таблицу.
- **Таблица бэкапов** (колонки: Имя файла, База данных, Размер, Дата создания, Действия):
  - Кнопка "Скачать" — `window.open()` на download endpoint.
  - Кнопка "Превью" — открывает модальное окно с загрузкой статистики.
  - Кнопка "Восстановить" — открывает модальное окно подтверждения.
- **Загрузка внешнего файла** — drag-and-drop или file input для `.dump` файлов с кнопками "Превью" и "Восстановить".

**Модальное окно "Превью бэкапа":**
- Заголовок: "Содержимое бэкапа"
- Секция "Источник": имя БД, дата бэкапа.
- Таблица статистики: Название таблицы | Количество записей.
- Кнопка "Закрыть" (если вызвано просто для просмотра).
- Кнопка "Перейти к восстановлению" (если вызвано перед restore).

**Модальное окно "Подтверждение восстановления":**
- Красное предупреждение: "Восстановление уничтожит ВСЕ текущие данные в базе `<db_name>`. После восстановления будут применены актуальные миграции. Эта операция необратима."
- Поле ввода: "Для подтверждения введите имя базы данных: `<db_name>`"
- Кнопка "Восстановить" заблокирована пока имя введено неверно.
- Прогресс-бар / спиннер во время восстановления (операция синхронная, тяжелая).

---

## 7. Безопасность

- **Восстановление** — самая опасная операция. Перед restore закрываются все соединения и терминируются активные сессии.
- **Временные БД для превью** создаются с уникальными именами и гарантированно удаляются после анализа (в `finally` блоке).
- **Доступ** — только admin (пока auth — stub).

---

## 8. Обработка ошибок

| Сценарий | Ответ |
|----------|-------|
| Бэкап не найден | 404 |
| Неверный формат файла (не pg_dump custom) | 422 |
| pg_restore вернул ошибку | 422 с stderr |
| Восстановление: имя БД введено неверно | 400 |
| Недостаточно прав (не admin) | 403 |
| Docker не доступен / контейнер не найден | 503 |
| Alembic upgrade failed после restore | 500 с логами alembic |
